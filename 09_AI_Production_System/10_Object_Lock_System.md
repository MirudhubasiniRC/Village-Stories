# Object Lock System

## Purpose

This permanent production manual keeps established Village Stories objects
identifiable, period-correct, materially credible, and continuous across shots.

It covers source packets, prompt construction, reference use, scene state,
Google Flow, QA, rejection, revision, handoff, and authority.

It does not authorise a new prop, substitute a modern equivalent, or establish
an unsupported object home.

## Core Rule

An object is not background decoration when it carries work, time, place,
relationship, sound, or continuity.

Generate the approved object in its approved state, use, and place.

## Authority

1. Approved current production decision.
2. `01_Canon/` continuity, timeline, and visual locks.
3. Relevant `04_Objects/` source document.
4. Relevant location and character canon.
5. Approved object reference or selected prior asset.
6. Approved script, board, and scene-state ledger.
7. This manual and production prompt system.
8. Tool defaults and candidates have no authority.

If source facts conflict, freeze the object state and seek a canon decision.

## Lock Vocabulary

- `OBJECT HARD LOCK`: established item identity.
- `MATERIAL HARD LOCK`: documented material and finish.
- `SCALE HARD LOCK`: plausible size relative to hand, body, room, or vehicle.
- `HOME LOCK`: normal approved storage or placement.
- `STATE FACT`: current condition and position.
- `HANDLER`: person or animal using the object.
- `VARIABLE`: approved temporary condition.
- `CARRYOVER`: prior shot's confirmed state.
- `NEGATIVE`: prohibited replacement or use.
- `OPEN`: undecided fact that blocks continuity-critical use.

## Object Source Packet

Build a packet for every story-relevant visible object.

```text
Episode / scene / shot ID:
Object name and category:
Why it is visible:
Owning source file and revision:
Approved object reference and role:
Identity and material locks:
Scale and weight impression:
Normal home / current location:
Current handler:
Start position / orientation:
Start condition:
Required action:
End position / orientation:
End condition:
Related character / location:
Sound implication:
Carryover source asset:
Permitted variables:
Required absences / negatives:
Open facts and owner:
```

Use exact object names from established object canon where available.

Do not use “traditional tool” when the specific approved tool matters.

## Packet Rules

- Load material before writing colour adjectives.
- Load scale before asking for a hand interaction.
- Load object home before placing a background prop.
- Load handler before describing an action.
- Load start and end state before creating motion.
- Record which hand holds an object.
- Record whether an object is open, closed, full, empty, hot, wet, clean, or dirty.
- Record whether it is on a surface, carried, worn, hung, stored, or absent.
- Record the destination when it moves.
- Record the next shot that inherits its state.

## Reference Hierarchy

1. Written object canon.
2. Approved dedicated object reference.
3. Approved material or construction research reference.
4. Selected previous asset for current state.
5. Approved source still or storyboard.
6. Generated output only after explicit approval.

Label reference role:

```text
OBJECT_IDENTITY
MATERIAL_CONSTRUCTION
SCALE_CONTEXT
OBJECT_HOME
HANDLING_POSE
INCOMING_STATE
DO_NOT_COPY
```

A material reference cannot establish a new object design.

A previous frame cannot excuse a known period error.

## Permanent Period Rule

All objects must be believable for early-1990s rural Tamil Nadu.

Prefer handmade, repaired, durable, practical materials and visible work.

Reject:

- Mobile phones.
- Screens.
- Digital-payment devices.
- LED fixtures.
- Modern kitchen appliances.
- Current branded packages.
- Plastic-dominant storage.
- Contemporary sports equipment.
- Modern vehicle interiors.
- Contemporary fashion accessories.
- Disposable modern drink containers.
- Modern power tools unless canon adds them.

## Immutable Object Facts

Lock these when applicable:

- Object identity.
- Purpose.
- Period.
- Primary material.
- Construction logic.
- Scale.
- Usual owner or location.
- Normal storage home.
- Safe handling method.
- Distinguishing marks or silhouette.
- Sound character when narratively relevant.
- Relationship to a fixed fixture.

## Permitted Variables

Record variables rather than treating them as redesign.

- Full or partly used food state.
- Water level.
- Heat or steam state.
- Wetness.
- Soil, ash, flour, turmeric, or grease.
- Open or closed lid.
- Hanging, carried, placed, or stored state.
- Light flame state for approved lamp.
- Tool orientation during a task.
- Temporary repair wear when source-backed.
- Cloth wrapping or cover when source-backed.

State changes need a visible cause or a ledgered off-screen transition.

## Kitchen Object Rules

Kitchen objects are work tools, not decorative clutter.

Select only the items needed for the action and location packet.

Keep vessel material, scale, heat, and placement believable.

Keep blades, stones, fire tools, and hot vessels safe and calm.

Keep storage items in their documented zones.

Do not add stainless-steel modern designs, plastic containers, non-stick pans,
gas hobs, mixers, refrigerators, or branded packaging.

## Cooking Vessels

Load `Cooking_Vessels.md` before showing a vessel as principal action.

Record:

- Vessel identity.
- Material.
- Heat state.
- Contents visibility.
- Lid state.
- Handle or rim handling.
- Stove or surface placement.
- Start and end position.

Do not change clay, brass, copper, iron, or other documented material casually.

Do not make a hot vessel float, bend, or switch scale between frames.

## Water Storage And Serving

Load the water-storage and serving source before showing water work.

Record source, vessel, fill level, carrying posture, destination, and final
placement.

Keep water motion subtle and physically plausible.

Do not turn ordinary water carrying into a spill, rescue, or spectacle beat.

## Grinding Stones

Grinding stones are heavy, stable, and location-bound.

Preserve their material, scale, surface, and placement.

Show only an approved, readable hand action.

Do not make the stone slide, bounce, rotate wildly, or appear lightweight.

## Cutting And Pounding Tools

Load the exact tool source before use.

Keep cutting and pounding grounded, cautious, and task-serving.

Do not create unsafe child handling.

Do not make tools weapons, threats, or action props.

## Fire And Measurement Tools

Fire requires a documented cooking or lighting purpose.

Keep flames small, controlled, and stationary relative to their source.

Keep smoke and heat appropriate to the action.

Do not add gas lighters, electric ignition, neon flame colour, or exaggerated
sparks.

## Food Storage

Storage objects must match kitchen storage logic and current use.

Record whether a container is closed, open, partly full, or being returned.

Do not add colourful commercial labels, plastic tubs, zip bags, or modern jars.

## Fibre And Wood Tools

Preserve woven, carved, or handmade material texture.

Keep wear functional rather than artificially distressed.

Do not replace fibre or wood with glossy plastic.

Do not add factory-perfect symmetry when handmade variation is expected.

## Brass Lamp

The Brass Lamp is an established religious lighting item.

Use it only in a documented Puja Corner, festival, devotional, or appropriate
location context.

Preserve brass material, modest scale, stable base, oil-flame logic, and
respectful handling.

Do not add electric bulbs, coloured LEDs, enormous flames, or a random lamp in
an unrelated room.

## Lanterns

Lantern use follows `Lighting/Lanterns.md` and approved time-of-day needs.

Preserve period construction, fuel-light logic, and warm restrained illumination.

Use lanterns to support visibility, never as a blue-black thriller device.

Do not replace with flashlights, electric torches, LEDs, or modern camping gear.

## Radio

The Radio is an established entertainment object.

Load its source before showing its design, placement, operation, or sound role.

Keep it period-correct and physically placed on a stable supported surface.

Do not add a remote control, cassette stack, modern speakers, screen, or
contemporary music interface.

## Bicycle

The Bicycle is an established vehicle object.

Load `Vehicles/Bicycle.md` before showing it prominently.

Keep its period-appropriate frame, practical condition, scale, resting method,
and route use.

At public locations bicycles lean against a wall, tree, or veranda rather than
occupying a modern rack.

Do not add gears, racing accessories, helmets, modern reflectors, or contemporary
branding unless source-backed.

## Bullock Cart

The Bullock Cart is an established vehicle.

Preserve wooden construction, practical scale, wheel behaviour, hitching logic,
resting-area relationship, and slow rural movement.

Do not modernise it, race it, overload it unrealistically, or use it as chase
machinery.

## Farming Tools

Load the established farming-tool source before use.

Record owner, crop or task, material, work state, carry posture, and storage
destination.

Keep tools practical, worn through normal use, and safely handled.

Do not invent mechanised equipment, weaponise tools, or use labour as spectacle.

## Furniture

Load `Furniture/README.md` and relevant location canon before showing furniture.

Furniture is fixed or heavy unless the scene specifically moves it.

Preserve material, scale, room placement, and practical use.

Do not add sofas, glass tables, plastic chairs, modular shelving, or hotel decor.

## Clothing And Toys

Clothing and toys require their owning canon and character packet.

They must not drift into generic modern props.

Record ownership, current user, condition, and storage home.

Do not add a toy merely to fill a child’s hands.

## Object Home Matrix

Maintain a current matrix:

```text
Object | Owner | Normal home | Current shot location | Handler | State |
Start | End | Next confirmed location | Source asset | Owner of update
```

Examples:

- Kitchen vessel: kitchen storage or work zone; moved only through a plausible task.
- Brass lamp: Puja Corner or approved festival context.
- Lantern: approved night-use location or documented storage.
- Bicycle: rider, house edge, or informal public leaning place.
- Bullock cart: route or resting area when scene-supported.
- Radio: stable supported domestic location when in use.

Do not leave the next confirmed location blank after a meaningful movement.

## Prompt Construction

Write object clauses in this order.

1. Exact object and source-backed material.
2. Current condition and location.
3. Handler and hand or support.
4. One visible action.
5. Start and end state.
6. Relation to relevant character and location.
7. Physical constraints.
8. Period and object-specific negatives.

Prefer observable facts over decorative adjectives.

## Object Prompt Template

```text
Use the established [OBJECT] made of [MATERIAL], at [SCALE] appropriate to
[HAND / BODY / LOCATION]. It begins [START STATE] at [START LOCATION], handled
by [HANDLER / HAND]. [HANDLER] performs [ONE ACTION], ending with the object
[END STATE] at [END LOCATION]. Preserve [OBJECT HOME / CARRYOVER FACT]. Do not
replace it with [MODERN OR UNSUPPORTED SUBSTITUTE].
```

## Water-Vessel Prompt Example

```text
Use the established traditional water-serving vessel in its approved material and
hand-scale. It begins full on the kitchen work surface. Father lifts it with both
hands and sets it at the courtyard-side threshold, upright and still full enough
for the next shot. Preserve its practical weight and the existing kitchen tool
homes. Do not turn it into plastic, a modern bottle, a steel thermos, or an
oversized decorative urn.
```

## Brass-Lamp Prompt Example

```text
At the established Puja Corner, use the approved modest brass lamp on its stable
base. Mother lights its small warm oil flame with calm deliberate handling; the
lamp remains upright in the same place. Preserve brass material, traditional
form, and restrained illumination. Do not add electric bulbs, coloured lights,
an enormous flame, or unsupported devotional objects.
```

## Bicycle Prompt Example

```text
Use the established period-correct bicycle leaning naturally against the Village
Square wall, front wheel turned slightly toward the path. It remains parked while
the approved character passes. Preserve practical scale and worn rural use. Do
not add a bike rack, modern racing parts, helmet, brand decals, or motor vehicle
features.
```

## Google Flow Rules

Flow may animate one approved object change.

It must not redesign, duplicate, melt, teleport, or exchange the object.

Use one-image input when the start state constrains the movement.

Use two-image input only when an approved endpoint materially differs.

State:

- Object identity.
- Handler.
- Principal action.
- Start state.
- End state.
- Permitted secondary motion.
- Fixed nearby objects.
- Negative transformations.

Example:

```text
Animate only Father lowering the same approved water vessel from both hands onto
the threshold. Keep the vessel material, size, fill level, upright orientation,
and final position unchanged. Keep all other kitchen tools, architecture, and
people still. No duplication, spill, deformation, new utensil, or modern object.
```

## Object Continuity Checks

Compare every object-critical asset with its predecessor.

- Same object identity.
- Same material.
- Same approximate scale.
- Same owner or handler.
- Same hand.
- Same start position.
- Same orientation.
- Same fill or heat state.
- Same lid or opening state.
- Same wear and cleanliness level.
- Same relation to fixture or surface.
- Same home when returned.
- Same period language.
- Plausible transition to next shot.

## Still QA

Approve only when:

- Object is source-backed.
- Object purpose is clear.
- Material is correct.
- Scale is correct.
- Placement is plausible.
- Handler is correct.
- Hands and contact are believable.
- Start state matches ledger.
- End state is visible or documented.
- Location supports its use.
- Period is correct.
- No unsupported object competes for attention.
- Object does not obscure a required face or action.

## Motion QA

Approve only when:

- Object retains identity every frame.
- Material does not change.
- Object count stays constant.
- Object does not morph.
- Object does not float.
- Handler contact remains credible.
- Weight and inertia feel plausible.
- Contents and flame behave plausibly.
- Start and end state match the ledger.
- No new object appears.
- No object vanishes.
- Camera does not conceal the required change.

## Rejection Categories

```text
OBJECT_ID_DRIFT
MATERIAL_DRIFT
SCALE_DRIFT
OBJECT_HOME_DRIFT
HANDLER_DRIFT
HAND_DRIFT
STATE_DRIFT
PERIOD_ERROR
PHYSICS_ERROR
FLOW_MORPH
UNSUPPORTED_SUBSTITUTE
UNSUPPORTED_ADDITION
```

Reject immediately for modern substitutes, incorrect material, impossible scale,
unlogged movement, changed hand, missing object, duplicate object, unsafe use, or
Flow deformation.

## Revision Protocol

Preserve the candidate, source packet, prompt, references, and QA result.

Name one primary failure category.

Correct the earliest responsible layer.

Check source packet completeness.

Check reference role.

Check start/end ledger.

Revise one prompt clause.

Regenerate a limited candidate set.

Recheck only the corrected defect and its neighbouring continuity facts.

If manipulation fails repeatedly, split the action into an insert or use two
approved still states rather than accepting a broken object.

## Handoff

```text
Asset ID and revision:
Shot ID:
Object and source file:
Object packet revision:
Reference files and roles:
Start state:
End state:
Handler and hand:
Object-home matrix update:
Incoming / outgoing asset IDs:
Flow input mode and action boundary:
QA result and rejection history:
Audio implication:
Crop-safe notes:
Approver and date:
```

Editors must preserve the visible order of object state changes.

Do not cut from a full vessel to an empty one, a parked bicycle to an absent one,
or an unlit lamp to a lit lamp without approved story logic.

## Change Control

This manual governs stable reproduction of existing objects.

It cannot create a new object class, product, material, vehicle, appliance,
religious item, tool, or period exception.

Create or revise the authoritative object canon first.

Then update object reference, location home, scene ledger, prompt module, and QA
criteria as one controlled change.

Never promote a generated prop to canon merely because it rendered well.

## Final Operator Check

- What exact object is this?
- Which source defines it?
- What material and scale lock it?
- Whose object is it?
- Where does it start?
- Who touches it?
- What visibly changes?
- Where does it end?
- What substitute must be excluded?

If one answer is unknown, hold the object-critical shot.

## Guiding Principle

The right object should feel as familiar and useful as the people and places that
share it, carrying the same lived history from shot to shot.
