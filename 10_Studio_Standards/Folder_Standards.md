# Folder Standards

## Purpose

This manual defines the permanent folder-governance rules for Village Stories.

It answers one operational question:

> Where must a governed item live so a future contributor can find, assess,
> reuse, review, and preserve it without guessing?

Folder structure is a continuity control.

It is not decoration, personal preference, or a substitute for ownership.

The purpose of a folder is to make a durable responsibility visible.

The purpose of a file path is to communicate custody and context.

The purpose of this manual is to prevent misplaced authority, duplicate truth,
lost source material, and accidental use of the wrong production output.

## Scope

This manual applies to folders and their contents in studio-governed storage.

It applies to repository paths, managed production storage, cloud workspaces,
and tool-specific libraries that hold governed work.

It applies to documents, prompts, media, project files, exports, manifests,
reviews, approvals, release packages, and archives.

It applies to humans, automation, and external tools that create or retain work.

It does not require every private thought or disposable experiment to become a
permanent studio record.

It does require any item that is shared, reviewed, selected, cited, reused, or
retained beyond its immediate task to have an accountable home.

This manual governs placement and folder responsibilities.

It does not define fictional truth, creative approval, prompt wording, media
quality, legal rights, or a tool’s operating steps.

Those matters remain with their designated authorities.

## Status

This is an active studio-wide governance manual.

It is intended to remain valid as production tools and storage providers change.

It must be read with the current approved version of related standards.

No local shortcut, tool default, or personal convention overrides this manual.

## Governing Principle

Place each governed item once in the narrowest durable home that owns its
responsibility, then reference it everywhere else.

When convenience conflicts with traceability, protect traceability.

When a copied file could silently diverge, use a reference instead.

When a folder is convenient today but confusing after handoff, choose the
structure that remains understandable after handoff.

When a physical storage limitation conflicts with this manual, preserve the
logical boundary in an index, manifest, or durable path reference.

## Core Commitments

Village Stories is a long-lived production system.

The folder system must survive changing personnel, tools, storage locations, and
release schedules.

The folder system must support both calm daily work and high-risk release work.

The folder system must make authority visible without copying authority.

The folder system must distinguish source evidence from production output.

The folder system must distinguish reusable assets from episode-specific use.

The folder system must distinguish candidates from selected material.

The folder system must distinguish current operational work from retained history.

The folder system must make cleanup deliberate rather than destructive.

## Relationship to Existing Top-Level Folders

This manual preserves the existing top-level repository folders exactly as they
are.

It does not authorize renaming any existing top-level folder.

It does not authorize moving existing canon, workflow, or asset material.

It does not prescribe a new top-level folder layout.

It does not treat examples in this manual as instructions to create missing
top-level folders.

Where another manual uses a logical storage illustration, that illustration must
not be used to restructure the established repository.

Existing project-level folder decisions remain in force until changed by their
own approved governance process.

If an example path conflicts with a real established path, retain the real path.

Record the mismatch as a documentation issue rather than moving material to make
the example true.

## Existing Top-Level Ownership Boundaries

The currently established top-level folders retain their existing ownership.

`00_Project_Core/` owns durable project identity and project-wide orientation.

`01_Canon/` owns approved permanent series rules and style decisions.

`02_World/` owns persistent setting, geography, culture, environment, and
location knowledge.

`03_Characters/` owns persistent character identity, behaviour, appearance, and
relationships.

`04_Objects/` owns recurring object definitions and their continuity.

`05_Research/` owns source evidence, research findings, cautions, and
interpretation records.

`06_Story_Engine/` owns story-construction systems and narrative logic.

`07_Episode_System/` owns episode form, pacing, audience structure, and
episode-design rules.

`08_Production/` owns production workflows, execution controls, and
production-method documentation.

`09_AI_Production_System/` owns AI-generation systems, prompt controls, locks,
and generation governance.

`10_Episodes/` owns per-episode packages, boards, generation, audio, edit, QA,
and release records for individual episodes.

`10_Studio_Standards/` owns shared studio conventions and governance manuals.

`11_Assets/` owns approved reusable assets and their durable indexes where
established.

`11_Templates/` owns approved fillable production forms (not doctrine).

`12_Lessons_Learned/` owns episode evidence and continuous-improvement registers.

`12_Quality_Assurance/` is a pointer shell only; operational QA authority lives
in Studio Standards, Production QC, and Prompt QA until migration retires it.

No folder becomes a lower-authority copy of another folder simply because a
production task needs convenient access.

## Authority Hierarchy

Apply the highest relevant authority when instructions overlap.

```text
Legal, rights, safety, and platform obligations
↓
01_Canon/ and approved canonical decisions
↓
02_World/ · 03_Characters/ · 04_Objects/ · 05_Research/
↓
06_Story_Engine/ · 07_Episode_System/
↓
10_Studio_Standards/ shared governance
↓
08_Production/ execution workflows and controls
↓
09_AI_Production_System/ generation controls and prompt systems
↓
Episode-specific approved records and managed asset indexes
↓
Drafts, experiments, candidates, and tool outputs
```

The hierarchy decides conflict resolution.

The hierarchy does not transfer folder ownership from one system to another.

An episode folder may cite a character record.

It may not become the permanent authority for that character.

A generated clip may be selected for one episode.

It may not become a reusable approved asset without the applicable asset review.

A release package may preserve a snapshot of metadata.

It may not replace the source record governing a later correction.

## Folder Ownership

Every durable folder must have one accountable owner role or owner system.

Ownership means responsibility for purpose, permissions, condition, and
retention.

Ownership does not mean that the owner personally performs every task inside it.

Ownership does mean the owner decides whether material belongs there.

The owner ensures the folder contains current navigation information when needed.

The owner ensures obsolete material is archived or marked without destroying
required history.

The owner ensures write access is limited to contributors who need it.

The owner ensures shared material has a review and handoff path.

The owner must not use folder ownership to override higher creative authority.

If no owner can be identified, the folder is not ready for governed use.

## Folder Stewardship Record

For a shared, long-lived, or high-risk folder, maintain a lightweight ownership
record in its index, README, manifest, or linked register.

The record should identify the folder purpose.

The record should identify the accountable owner role.

The record should identify the normal contributors or write roles.

The record should identify the authoritative upstream sources.

The record should identify the retention expectation.

The record should identify the escalation route for placement disputes.

Do not create administrative records for a one-hour private scratch folder.

Create them when a folder becomes a handoff boundary.

## Read Authority

Read authority determines who may consult material for work.

Approved source material should be readable by roles that need it for accurate
production.

Read access to rights-sensitive, personal, contractual, or unreleased material
must be limited to the roles with a legitimate need.

Read access does not grant permission to copy, modify, publish, or reuse.

An operator may read an approved visual reference without becoming its owner.

A reviewer may read a release package without being permitted to alter it.

Where platform permissions cannot express the desired rule, enforce it through a
documented owner and access process.

## Write Authority

Write authority determines who may add, modify, move, or delete material.

Writers may add work only to folders whose purpose permits that artifact.

Writers may not modify higher-authority source records merely to solve a local
episode problem.

Writers may not overwrite approved media with a newer export.

Writers may not delete an item subject to review, release, rights, or retention
obligations.

Writers may create a controlled working subfolder only when its owner and
retention are clear.

Write access to a shared folder should be the minimum needed for safe handoff.

Where permissions are broad, reviewers must apply the same restraint manually.

## Approval Authority

Folder placement never substitutes for approval.

An item in an `approved`-named container is not approved unless a named
approval record supports it.

An item in a release package is not authorized for publication unless the
release gate is complete.

An item in an archive is not automatically invalid as historical evidence.

An item in a source folder is not automatically accurate or current.

Approval comes from the applicable content owner and documented review process.

## Folder Names

Folder names must follow `10_Studio_Standards/Naming_Standards.md`.

Use lower-case ASCII filesystem names.

Use durable purpose names rather than temporary activity names.

Use an ordering prefix only when sibling order is meaningful.

Use a hyphen inside a token when multiple words are needed.

Avoid spaces, emojis, private initials, and vague emotional labels.

Do not name a folder `new`, `latest`, `misc`, `other`, `final`, or `stuff`.

Do not name a permanent folder after a temporary person, session, or tool mode.

Do not use dates as a substitute for purpose.

Use a date only when the folder represents a dated event, snapshot, or intake.

## Folder Depth

Use the shallowest structure that keeps ownership and state unambiguous.

Do not create a subfolder merely because a file type exists.

Create a subfolder when it separates a distinct responsibility, access rule,
state boundary, retention rule, or handoff.

Avoid paths so deep that contributors cannot understand the purpose of each
level.

Avoid wide dumping folders that require opening every file to identify its role.

As a default, a contributor should understand an item’s domain, work scope, and
state from its path and name.

If a folder needs more than one screen of unrelated files, assess whether a
functional subdivision is needed.

Do not split a small folder merely to imitate a template.

## Folder Creation Rule

Create a permanent folder only when all of the following are known:

1. Its durable purpose.

2. Its accountable owner.

3. Its expected contents.

4. Its upstream authority or source relationship.

5. Its read and write boundary.

6. Its retention or cleanup rule.

7. Its relationship to existing sibling folders.

Temporary folders may be created with a lighter rule.

They still require a task owner, an expiry, and a place that permits temporary
work.

Do not create a permanent-looking folder for a one-time experiment.

Do not create a temporary folder inside an authoritative source folder.

Do not create a folder to avoid resolving where the item actually belongs.

## New Subfolder Decision Test

Before creating a subfolder, answer these questions.

Does it hold one coherent type of work?

Does it have a different owner, state, access rule, or retention rule?

Will it remain meaningful after the immediate task ends?

Can a future contributor infer its purpose from its name?

Does an existing folder already own this material?

Can a manifest, filename, or tag solve the problem more simply?

If the answer to the final two questions is yes, do not create the subfolder.

## File Placement Rule

Place a file in the folder that owns the file’s primary responsibility.

Do not place a file based only on the application used to create it.

Do not place a file based only on the person who created it.

Do not place a file based only on the most recent episode that used it.

Place a reusable prompt with its prompt system or reusable prompt library.

Place an episode-specific prompt instance with the episode generation record.

Place a canonical appearance rule with the character authority.

Place a selected episode still with the episode record or managed production
asset storage linked from it.

Place release metadata with the release package and preserve the approved source
record it derives from.

## One Authoritative Home

Every governed source file has one authoritative home.

Every governed media source has one authoritative storage location or asset
record.

Every approved release master has one identified immutable master.

Every reusable asset has one asset identifier and one governing register.

Every selected episode asset has one selection record.

Other locations may contain links, manifests, proxies, exports, or snapshots.

Other locations must not contain unlabelled competing sources.

If two copies appear equally current, stop and resolve the conflict before use.

## Links Instead of Copies

Use a path reference, URL, asset identifier, manifest entry, or shortcut when
another system needs to locate authoritative material.

A reference should identify the source path or storage URI.

A reference should identify the relevant version, revision, or immutable ID.

A reference should identify the purpose for which the source is being used.

A reference should identify any scope limitation, such as one episode only.

Do not copy a canonical excerpt into a production folder as a substitute for a
reference.

Do not duplicate a reusable prompt so it can be edited locally without lineage.

Do not copy a release master into multiple “final” folders for convenience.

## Permitted Copies

A copy is permitted only when its purpose is explicit.

Permitted copy purposes include review proxy, editorial working copy, platform
upload copy, restoration backup, legal delivery copy, and immutable release
snapshot.

Each permitted copy must identify its relation to the source.

Each permitted copy must identify whether it may be edited.

Each permitted copy must identify its expiry or retention rule when temporary.

If its source changes, do not silently treat the copy as updated.

Create a new derived copy or explicitly reconcile it.

## Forbidden Duplication

The following duplication is prohibited.

Do not maintain two editable “master” scripts for one episode.

Do not keep copied canon pages in episode folders as working truth.

Do not duplicate character, location, object, or style locks across prompt
folders without a source reference.

Do not keep identical generated media under several untracked filenames.

Do not copy a selected asset into a reuse library without approval and provenance.

Do not create parallel release packages with ambiguous status.

Do not export a platform download to a second permanent source folder without
recording the derivation.

Do not duplicate a prompt merely to change a single shot field.

Do not preserve personal desktop copies as hidden production sources.

## Source, Derived, and Archive Separation

Every governed item belongs to one of three custody classes.

The custody class must be inferable from folder purpose, file status, manifest,
or both.

The three classes are source, derived, and archive.

These classes describe relationship and use.

They do not by themselves decide creative quality or approval.

## Source Material

Source material is the original authoritative input or recorded evidence from
which work is made or judged.

Examples include approved canon documents, research evidence, licensed source
media, original recordings, authored scripts, approved prompt modules, and
source project files.

Source material must preserve provenance.

Source material must not be casually overwritten by an export.

Source material must not be replaced by a proxy because the proxy is easier to
open.

Source material may be revised through its owning process.

Source material must identify its source relationship when imported from outside
the studio.

## Derived Material

Derived material is made from one or more source items.

Examples include rendered videos, image exports, audio mixes, flattened PDFs,
review proxies, generated candidates, thumbnails, captions, and upload files.

Derived material must identify its source or source register where practical.

Derived material must never silently replace the source project or source media.

Derived material may be disposable when it is not selected, reviewed, or needed
for evidence.

Derived material selected for an episode requires a durable selection record.

Derived material selected for release requires release-package custody.

## Archive Material

Archive material is retained history that is not current operational input.

Examples include superseded drafts, withdrawn candidates, prior approved
milestones, closed reviews, released package snapshots, and retired references.

Archive material must remain readable enough to understand its context.

Archive material must identify why it was retained.

Archive material must identify its replacement or historical relationship where
applicable.

Archive material must not be silently returned to active use.

Archive material must not be edited except for approved preservation metadata.

## Source Folder Rules

Store source material in the owning authoritative folder or source repository.

Keep source project files separate from exports when both are retained.

Keep original media separate from normalized, compressed, or editorial copies.

Keep imported reference media separate from studio-made reference assets.

Keep evidence of rights, origin, and license with the source record or asset
register.

Never replace an original with a colour-corrected, cropped, captioned, or
watermarked derivative.

Do not change a source filename solely to make it resemble a derived filename.

## Derived Folder Rules

Derived work must live in a folder that states its production purpose or state.

Separate working renders from approved deliverables.

Separate candidate media from selected media.

Separate edit proxies from source-quality media when both exist.

Keep a manifest when many derived files map to a small number of source items.

Do not retain every automatically generated intermediate by default.

Retain the intermediates required for reproducibility, review, rights, or a
known future workflow.

Delete or expire the rest according to the temporary-file rule.

## Archive Folder Rules

Archive only after the active successor or final disposition is clear.

Move as a deliberate historical action, not as a way to hide confusing work.

Record the archive reason.

Record the archived item’s former active path when useful for recovery.

Record the successor path or release identifier when applicable.

Preserve immutable release packages and material approval evidence.

Do not archive unresolved work merely because the schedule changed.

Unresolved work remains active, paused, or rejected until its disposition is
recorded.

## Episode Workspaces

An episode workspace is a bounded work area for one stable episode identifier.

Use the project-assigned episode identifier.

Do not create a competing episode numbering system.

An episode workspace may live in established episode storage or approved managed
production storage.

This manual does not create or mandate a new top-level episode folder.

The workspace must link back to the episode’s governing brief and records.

The workspace must link to authoritative canon, world, character, object, and
research sources rather than copy them.

The workspace must preserve the path to selected assets and release evidence.

## Episode Workspace Functional Pattern

When a bounded episode workspace is needed, use these logical functions.

```text
[episode-id]/
  00_admin/
  01_story/
  02_references/
  03_storyboard/
  04_generation/
  05_audio/
  06_edit/
  07_qa/
  08_release/
  09_archive/
```

This is a logical pattern, not authorization to restructure the repository.

Use it only inside an established episode-work location or an approved external
production workspace.

An equivalent external layout is allowed if its index preserves these boundaries.

Do not create empty functional folders merely to complete the pattern.

Create only the functions used by the episode.

## Episode Admin Area

The logical `00_admin/` area holds administrative coordination.

It may hold the brief, owner list, schedule, status board, approvals, and
handoff notes.

It must not become a second location for source scripts, media masters, or
canon.

It should identify the current episode scope and responsible roles.

It should point to the selected current revision of each major work product.

It should record escalation decisions that affect downstream work.

## Episode Story Area

The logical `01_story/` area holds episode-specific premise, outline, script,
dialogue, narration, scene list, and story review records.

It must not contain permanent story-system guidance copied from `06_Story_Engine/`.

It must not contain duplicate episode-format rules copied from
`07_Episode_System/`.

Use references to those systems when their rules constrain the episode.

Keep only episode-specific application and decision records here.

## Episode References Area

The logical `02_references/` area holds the episode reference manifest and
permitted working reference copies.

The manifest must point to authoritative canon, world, character, object, and
research paths.

It must state the applied fact or decision, not merely list a broad folder.

Reference copies must state whether they are temporary, licensed, or approved.

Do not put an editable replacement for a source authority in this area.

Do not collect unverified internet media here without provenance and usage notes.

## Episode Storyboard Area

The logical `03_storyboard/` area holds episode-specific boards, shot lists,
animatic notes, shot contracts, and camera continuity records.

It should preserve the current approved storyboard revision.

It should link selected keyframes rather than duplicate reusable asset sources.

It must not hold permanent camera-language authority copied from canon or the
AI production system.

It may hold the episode-specific interpretation of those authorities.

## Episode Generation Area

The logical `04_generation/` area holds episode prompt instances, generation
records, input-reference manifests, candidate registers, and selected-shot
records.

It may contain approved source inputs required to run an episode generation task
when access rules permit.

It must identify whether inputs are source, temporary working copies, or
derived candidates.

It must preserve prompt lineage to the reusable prompt or module source.

It must not become a second permanent prompt library.

It must not contain untracked downloads from generation tools.

## Episode Audio Area

The logical `05_audio/` area holds episode-specific dialogue, voice, ambience,
Foley, music-use records, mixes, stems, and audio review notes.

Keep recordings and source sessions separate from compressed previews and final
mixes.

Keep rights and provenance records linked to licensed or external audio.

Do not place a reusable approved sound asset here merely because one episode
uses it.

Reference the reusable asset identifier and record its episode use.

## Episode Edit Area

The logical `06_edit/` area holds edit project files, sequences, proxy maps,
subtitle working files, review exports, and edit notes.

Separate editable projects from rendered review exports.

Separate review exports from approved picture and sound masters.

Do not make an editor’s local cache an authoritative source.

Do not retain every automatic render if a review manifest identifies the needed
versions.

## Episode QA Area

The logical `07_qa/` area holds checklists, issue logs, continuity findings,
technical checks, review evidence, waivers, and release decisions.

QA records must cite the reviewed asset or revision precisely.

QA records must not be stored only in chat, email, or an unlinked platform
comment thread.

Closing an issue must not delete the original finding.

Record the disposition, corrective work, reviewer, and verification.

## Episode Release Area

The logical `08_release/` area holds the package prepared for one platform
release.

It contains approved deliverables and the records needed to verify the delivered
release.

It must distinguish the immutable approved master from platform-specific copies.

It must identify the exact thumbnail, captions, metadata, credits, and approval
evidence associated with the release.

It must not contain alternate unapproved masters that could be uploaded by
mistake.

It must not become the sole home of reusable thumbnail components or music.

## Episode Archive Area

The logical `09_archive/` area holds episode-scoped retained history.

It may hold superseded drafts, withdrawn candidates, frozen milestones, and
closed review evidence.

It must not be used as a temporary trash bin.

It must not contain active work without an explicit restoration or reopening
decision.

It must preserve enough context to explain why an item was archived.

## Work Folder Pattern

A work folder is a bounded area for a defined task, change, or production stage.

Use a work folder only when the task has more than one governed item or requires
a controlled handoff.

Name it by scope and purpose, not contributor name.

Examples of valid logical work-folder names include:

```text
ep-014__shot-repair/
ep-014__release-preflight/
series__thumbnail-refresh/
character-mother__reference-review/
```

The work folder must identify an owner and expected closing condition.

The work folder must either resolve into the authoritative home or archive its
retained evidence at closure.

Do not leave a completed work folder indefinitely marked as active.

## Work Folder Lifecycle

At creation, record the task, owner, source inputs, expected output, and expiry
or close condition.

During work, keep drafts and temporary material within the work boundary.

At review, identify the exact candidate or revision under consideration.

At approval, move or register the approved output in its permanent owned home.

At closure, archive required evidence and remove disposable temporary material.

At reopening, record why the work is active again and which prior decision is
being reconsidered.

## Asset Folder Pattern

An asset folder is for durable reusable material, not merely a file with visual
or audio content.

An asset must have a stable identity before it is treated as reusable.

The asset record must identify provenance, rights, owner, status, and permitted
use.

Separate source media, editable masters, approved deliverables, previews, and
archive history.

A logical asset pattern may be:

```text
[asset-id]__[asset-purpose]/
  source/
  working/
  approved/
  derivatives/
  records/
  archive/
```

This pattern is a custody model.

Adapt it only within an established asset store or managed external storage.

Do not create a competing reusable-asset library in an episode workspace.

## Asset Source Area

The asset `source/` area holds original capture, licensed source, original
project file, or imported reference as applicable.

Preserve original provenance and right-to-use evidence.

Do not edit in place unless the source format itself is the approved editable
master and version control is active.

Do not store a flattened export as the only source when an editable master is
required for future use.

## Asset Working Area

The asset `working/` area holds controlled edits that are not approved reusable
assets.

Keep work that changes identity, crop, colour, content, or rights treatment
separate from approved material.

Apply temporary-file expiry rules to disposable working products.

Do not distribute from this area without explicit review.

## Asset Approved Area

The asset `approved/` area holds assets approved for the declared reuse scope.

Approval must state the scope, such as series-wide, location-specific, episode
limited, editorial only, or platform specific.

The presence of a file in this area does not license use outside its stated
scope.

Do not place multiple unlabelled alternatives in this area.

Use an asset register to state which revision is active.

## Asset Derivatives Area

The asset `derivatives/` area holds exports made from approved or source assets.

Examples include web-sized previews, crops, thumbnails, audio transcodes, and
platform encodes.

Each derivative must preserve its source relationship.

Do not elevate a derivative to an approved master without its own review.

Do not keep derivatives in the source area.

## Reference Media

Reference media is material consulted to inform work.

It is not automatically a reusable studio asset.

Reference media may be visual, audio, textual, historical, cultural, technical,
or platform-specific.

Store reference media with a reference manifest or in the owning research,
canon, asset, or episode-reference context.

Record origin, access date when relevant, permitted use, and applied purpose.

Record whether the item is evidence, inspiration, a visual match target, or a
licensed production input.

Do not treat a search result, screenshot, or borrowed image as free-to-use
production media.

Do not bury rights-sensitive reference media in a generic downloads folder.

## Reference Media Placement Rules

Place durable research evidence under `05_Research/` or its approved linked
storage.

Place permanent visual style references under the authority that owns the style
decision.

Place character-specific appearance references under the relevant character
authority or approved asset register.

Place location-specific references under the world or location authority.

Place episode-only working reference copies in the episode references area with
their source and expiry.

Place externally licensed production inputs with their asset record and rights
evidence.

Do not duplicate source reference media across all episodes that cite it.

## Prompt Placement

Prompts are controlled production records, not disposable tool text.

Place reusable master prompts and prompt modules in `09_AI_Production_System/`
or their designated prompt-library location.

Place lock systems with the AI production system that owns their maintenance.

Place episode-specific prompt instances in the episode generation area.

Place a selected generation prompt beside or linked from the selected-shot
record.

Place prompt QA evidence with the episode QA record or designated prompt-QA
record.

Do not place master prompts inside an episode folder merely because one episode
uses them.

Do not place copied lock text in a prompt instance without a source version
reference.

## Prompt Lineage Rule

Every episode prompt instance must identify its reusable source prompt or module.

Every prompt revision must identify its episode, scene, shot, or bounded task.

Every selected generated output must identify the prompt revision used.

Every prompt source must identify the owning system and approval status.

When a prompt is forked for an experiment, record that it is experimental.

An experimental fork must not silently replace the reusable source prompt.

If the fork becomes reusable, review and promote it through `09_AI_Production_System/`.

## Google Flow Project Organization

Google Flow is a production environment, not the studio’s sole source of truth.

Create a Flow project for one episode or a clearly bounded sequence.

Use the approved project naming convention from the Google Flow workflow.

Keep the durable project reference in the episode generation record.

Record the Flow project or task identifier, exact prompt revision, reference set,
model or mode where known, duration, aspect ratio, output, and review decision.

Export or preserve selected material according to rights, storage, and tool
availability rules.

Do not rely on browser history, a personal login, or a Flow workspace label as
the only path to selected work.

## Google Flow Inputs

Keep Flow inputs classified before upload.

Approved start frames and end frames must link to their asset IDs or approved
paths.

Character, location, and object references must point to approved source
records.

Temporary resized upload copies must remain temporary and must not replace the
source-quality reference.

Record a failed or rejected output in the candidate register when it informs a
material generation decision.

Do not upload unverified media as an ingredient simply because it appears
visually convenient.

## Google Flow Outputs

Treat each downloaded Flow output as a candidate until selected and approved for
its stated purpose.

Store candidates in the episode generation candidate boundary or managed
equivalent.

Do not rename a selected output in a way that erases its Flow task relationship.

Record the selected output in the shot record.

Move or copy only the approved editorial derivative into the edit boundary.

Preserve the original exported candidate when needed to reproduce or defend the
selection.

Do not store Flow cache files, duplicate downloads, or browser captures as
permanent studio assets without a documented purpose.

## YouTube Release Package Organization

Each YouTube release requires one bounded release package.

The package must be associated with one episode identifier and one intended
platform release.

The package must be assembled only after its components are reviewable.

The package must distinguish editable sources, approved upload deliverables, and
immutable release evidence.

A logical release-package pattern may be:

```text
[episode-id]__youtube-release/
  00_manifest/
  01_master/
  02_thumbnail/
  03_metadata/
  04_captions/
  05_rights-and-credits/
  06_platform-proof/
  07_approval/
  08_archive/
```

This pattern is logical and may be represented in existing storage through a
manifest and linked controlled paths.

It does not authorize creation or restructuring of a top-level release folder.

## Release Manifest

The logical `00_manifest/` area identifies the complete package.

The manifest must list the episode ID.

The manifest must list the intended YouTube release state.

The manifest must list the exact master, thumbnail, captions, and metadata
revisions.

The manifest must list the applicable QA authorization and any approved waiver.

The manifest must list the package owner and target publication time when set.

The manifest must list the public URL after publication.

The manifest is the package index, not a substitute for the governed source
files.

## Release Master Area

The logical `01_master/` area contains the approved upload master and any
strictly necessary platform delivery derivative.

Keep the immutable approved master clearly identifiable.

Keep test encodes, review exports, and unapproved alternates outside this area.

Record the master’s source project and approval record.

Use a checksum when feasible for a high-risk release master.

Do not alter or replace a release master after approval without reopening the
required gates.

## Release Thumbnail Area

The logical `02_thumbnail/` area contains the exact approved upload thumbnail,
its version identifier, and linked source record.

Do not store a general library of reusable thumbnail components as if they were
episode-release deliverables.

Do not upload a thumbnail candidate because it is nearby in a workspace.

Keep the selected upload image distinguishable from editable design sources.

## Release Metadata Area

The logical `03_metadata/` area contains the approved title, description,
chapters, playlist choice, language fields, cards, end-screen plan, links, and
pinned comment if used.

Use a controlled record rather than relying on text currently visible in the
YouTube interface.

Record the final entered platform values after upload.

Do not allow a copied metadata draft to silently become public copy.

Keep corrections as a documented revision rather than overwriting history.

## Release Captions Area

The logical `04_captions/` area contains approved captions, subtitle files,
language variants, and accessibility review evidence.

Keep the uploaded caption file identifiable by language and revision.

Keep machine-generated caption candidates separate from reviewed captions.

Do not assume the platform’s automatic caption track is the approved studio
caption record.

## Release Rights and Credits Area

The logical `05_rights-and-credits/` area contains required rights, credits,
licenses, disclosures, and audience-suitability decisions.

Restrict access where contracts or personal information require it.

Do not put passwords, tokens, private account recovery data, or credentials in
the release package.

Reference secure credential management rather than storing secrets in files.

## Platform Proof Area

The logical `06_platform-proof/` area contains evidence that the exact private,
scheduled, and public YouTube release was checked.

It may contain screenshots, exported records, review notes, public URL records,
and platform-processing observations.

It must identify the observation date and reviewer.

It must not be treated as the editable source of metadata or assets.

## Release Approval Area

The logical `07_approval/` area contains release authorization, QA disposition,
approved waivers, and named owner decisions.

The approval must name the item or exact revision approved.

The approval must state whether it is conditional.

The approval must not be implied by moving a file into this area.

## Temporary Files

Temporary files are useful only while their task remains active.

They include caches, previews, transcoding intermediates, test exports,
scratch notes, duplicate downloads, auto-saves, failed uploads, and tool
artifacts.

Temporary files must be stored only in a folder that permits temporary work.

Temporary files must have an owner and task relationship.

Temporary files must not be placed in authoritative source, approved, or release
master areas.

Temporary files must not be used as a handoff source without promotion and
registration.

Temporary files must be deleted or deliberately archived when their retention
need ends.

## Temporary Folder Pattern

When a temporary folder is necessary, name it for task and expiry.

Use an explicit temporary marker permitted by the local workflow, such as:

```text
tmp__ep-014__flow-downloads__expires-2026-08-01/
```

The marker does not excuse ambiguous storage.

The folder must state where approved outputs will be promoted.

The folder must be reviewed at task close.

Do not keep a `tmp` folder indefinitely as a second asset library.

## Temporary File Promotion

Before promoting a temporary item into governed storage, verify its source.

Verify its identity and naming.

Verify its review state.

Verify its owner and reuse scope.

Verify its relationship to source inputs.

Verify that an equivalent approved item does not already exist.

Move it to the appropriate permanent home or register the managed storage path.

Do not promote an entire temporary folder without classifying its contents.

## Downloads

Downloads are unclassified intake, not permanent storage.

Review downloaded material before moving it into a governed folder.

Capture origin, download date when material, version or task ID, and intended
use.

Delete duplicate and unusable downloads after required evidence is retained.

Do not reference an item from an ungoverned downloads location in a permanent
manifest.

Do not use a download filename as the permanent studio filename.

## Intake and Quarantine

Use an intake or quarantine boundary when external material cannot be classified
immediately.

Quarantined material must not be treated as approved source material.

Record origin, rights status, sender or system, receipt date, and intended use.

Review for rights, relevance, safety, malware risk, and placement.

Then classify it as source, derived, reference, temporary, or rejected.

Do not leave external material permanently unclassified.

## Archive Retention

Retain material when it supports historical truth, legal obligations, rights,
approval evidence, reproducibility, or recovery.

Retain immutable release masters and package evidence.

Retain approved source records and their meaningful history.

Retain selected generated outputs and generation records to the degree required
to explain the release.

Retain rejected candidates only when their record prevents repeated failure or
supports a material decision.

Do not retain large disposable intermediates merely because storage is available.

Apply any legal or contractual retention rule before ordinary cleanup.

## Version Control Relationship

`08_Production/17_Version_Control.md` governs history, revisions, approval
records, recovery, and safe change practice.

This manual governs where those controlled items live.

Use Git or an equivalent change history for suitable text documentation.

Do not use Git history as an excuse for uncontrolled folder placement.

Do not commit cache files, generated previews, personal settings, duplicate
downloads, or disposable binary exports merely because they exist.

Use stable asset identifiers, manifests, managed storage, and release archives
for large media where Git is unsuitable.

Never overwrite the only approved source or master because history exists
somewhere else.

## File Organization Relationship

`08_Production/16_File_Organization.md` governs production filing, statuses,
source references, and the detailed execution model.

This manual sets the studio-wide folder policy that supports it.

When the production manual defines a more specific placement for a production
artifact, follow that specific rule within its declared scope.

When it uses a logical model, do not use that model to rename or restructure
existing repository canon.

Resolve any apparent conflict by retaining the established path and escalating
the documentation mismatch to the relevant owners.

## Naming Standards Relationship

`10_Studio_Standards/Naming_Standards.md` is authoritative for folder syntax,
characters, token order, case, and prohibited naming patterns.

This manual is authoritative for folder purpose, placement, custody, and
ownership boundaries.

Use both standards together.

A correctly named folder can still be misplaced.

A correctly placed folder can still be improperly named.

Neither condition makes a file authoritative without the applicable approval.

## 08 Production Relationship

`08_Production/` owns methods for executing production work.

Production workflows decide how a task proceeds through story, storyboard,
generation, sound, edit, QA, and release.

This manual requires those work products to remain in distinguishable functional
and custody boundaries.

Do not move a production workflow into this standard.

Do not copy this governance manual into each production workflow.

Cross-reference the governing document and apply the placement rule.

## 09 AI Production System Relationship

`09_AI_Production_System/` owns master prompts, prompt modules, locks,
generation controls, prompt QA, and tool-specific AI production guidance.

This manual protects their placement and lineage.

Reusable AI controls remain in their AI system home.

Episode-specific applications remain with the episode generation record.

Selected outputs remain traceable to both their prompt lineage and source
references.

Google Flow convenience does not change canon authority or folder ownership.

## Continuity Relationship

Folder governance protects continuity by preserving source relationships.

Character, location, object, and style authority remain with their owning
systems.

An episode workspace records how those authorities were applied.

It does not rewrite them by copying a local summary.

When a local production decision conflicts with a source authority, stop and
escalate.

Do not “fix” the discrepancy by changing only a duplicate local file.

## Folder Quality Assurance

Perform folder QA at every major handoff.

Check that each item is in the folder that owns its primary responsibility.

Check that source, derived, and archive material are visibly separated.

Check that reusable items are not trapped in an episode-only workspace.

Check that episode-only items are not misrepresented as reusable assets.

Check that prompts link to their source modules and generated outputs.

Check that selected assets point to review and approval evidence.

Check that release packages identify the exact upload deliverables.

Check that no credentials, secrets, or personal sensitive data are stored in
governed release or repository files.

## Folder QA Questions

Can a new contributor locate the authoritative source without asking its author?

Can a reviewer distinguish the source from a derived export?

Can an editor identify the selected asset without opening every candidate?

Can a Flow operator find the exact prompt and reference set for a selected shot?

Can a publisher identify the one approved upload master and thumbnail?

Can a future correction locate the released metadata and public URL record?

Can an archivist explain why retained material is no longer active?

Can a contributor identify the owner of a placement dispute?

If any answer is no, correct the path, manifest, or ownership record before
handoff.

## Handoff Minimums

Before handing off a governed folder, provide a concise orientation record when
the next role cannot infer the needed context from the files alone.

State the work scope.

State the current approved or selected revision.

State unresolved questions and their owner.

State the source paths or asset IDs relied upon.

State what the receiving role may change.

State what must not be altered without review.

State the expected return path or next handoff.

Do not hand off a folder full of files with a message that says only “please
check.”

## Examples

### Example: character reference used in an episode

The authoritative appearance record remains under `03_Characters/`.

The episode reference manifest points to the exact appearance section used.

The selected reference image is linked by asset ID or controlled path.

The episode does not receive an editable copied character profile.

This preserves character continuity while enabling production access.

### Example: reusable prompt used for one shot

The master prompt remains in `09_AI_Production_System/`.

The prompt module remains in its owning prompt library.

The shot-specific instance is stored with the episode generation record.

The shot record cites the module revision and all added shot constraints.

The selected clip cites the exact instance that produced it.

This preserves prompt lineage without treating the episode instance as a master.

### Example: Flow candidate to selected edit clip

A Flow download first enters the episode generation candidate boundary.

Its task identifier, prompt revision, references, and output name are recorded.

After review, one output is selected in the shot record.

An editorial derivative may be placed in the edit boundary.

The original candidate relationship remains traceable.

The editorial derivative does not overwrite the candidate export.

### Example: YouTube release

The approved upload master is identified in the release package.

The exact thumbnail, captions, metadata, credits, and QA authorization are
listed in the package manifest.

The private-upload verification is retained as platform proof.

The published URL and final entered metadata are recorded.

Later corrections create a documented revision rather than erase release history.

## Non-Examples

### Non-example: canon copy in production

Copying `03_Characters/` material into a new episode folder and editing it there
is prohibited.

It creates a competing authority and risks silent divergence.

Use a reference manifest and escalate any needed canon change.

### Non-example: final-final exports

Keeping `final.mp4`, `final-final.mp4`, and `final-use-this.mp4` in one folder
is prohibited.

The names do not establish approval or source relationship.

Use a controlled revision, selected state, release package, and approval record.

### Non-example: personal ownership path

Saving shared production work only under a contributor-named private folder is
prohibited.

Personal working space may be temporary.

Shared governed material must return to its owned studio location before handoff.

### Non-example: duplicate reusable asset

Copying a recurring ambience loop into every episode audio folder and treating
each copy as the source is prohibited.

Maintain the reusable asset once and record each episode’s permitted use.

### Non-example: untracked platform source

Relying on a Google Flow project URL or YouTube draft as the only source record
is prohibited.

Platforms change, access changes, and browser history is not durable governance.

Maintain a studio record with identifiers and controlled exports.

## Exceptions

An exception is a time-bounded approved departure from this manual.

An exception must solve a real constraint, not a preference.

An exception must not silently establish a new studio convention.

Request an exception before the departure when practical.

For urgent release, safety, rights, or recovery work, record the exception as
soon as the immediate risk is controlled.

## Exception Record

Every exception record must state the rule affected.

It must state the folder or asset scope.

It must state the reason the normal rule cannot be followed.

It must state the risk introduced.

It must state compensating controls.

It must state the accountable approver.

It must state a review or expiry date.

It must state the restoration or migration plan.

An expired exception is not continuing permission.

## Acceptable Exception Examples

A platform-required folder name may differ from studio syntax when the external
platform cannot accept the standard name.

A legal delivery may require a duplicate package in a prescribed external
structure.

A recovery event may require a temporary read-only restoration folder.

A collaboration partner may require a controlled shared copy outside studio
storage.

In each case, retain a manifest linking the exception location to the
authoritative studio record.

## Migration

Migration is a controlled change of storage path, folder structure, provider, or
custody boundary.

Do not migrate merely to make paths appear cleaner.

Do not use a migration to rename or restructure canon without the relevant
authority.

Plan migration when storage risk, access loss, scale, tool retirement, or a
documented governance improvement requires it.

Preserve source relationships, approvals, revision history, and release records.

## Migration Preparation

Identify the source and destination boundaries.

Identify folder owners and approvers.

Inventory governed files, asset IDs, external links, and manifests.

Classify source, derived, temporary, and archive material.

Identify items that must remain immutable.

Identify rights, privacy, or platform constraints.

Identify all documents that cite affected paths.

Create a rollback plan before moving high-risk material.

## Migration Execution

Move or copy in controlled batches.

Verify item counts, checksums when feasible, names, and permissions.

Update manifests and references only after destination verification.

Record old-to-new path mappings for durable or frequently cited material.

Do not delete the source boundary until recovery and approval conditions are met.

Do not alter content merely because its path changes.

Do not merge folders with different ownership without an approved ownership
decision.

## Migration Completion

Confirm that authoritative paths resolve.

Confirm that release-package references remain intact.

Confirm that episode and asset indexes point to the destination.

Confirm that access permissions match the intended read/write model.

Confirm that temporary staging copies are cleaned up.

Confirm that the old path has a migration notice, archive marker, or approved
retirement record as appropriate.

Document material exceptions and unresolved follow-up work.

## Placement Disputes

Resolve a placement dispute by asking which system owns the item’s primary
responsibility.

If the item expresses permanent fact, use the authority that owns that fact.

If it expresses episode-specific application, use the episode work boundary.

If it is a reusable production method, use the owning workflow or AI system.

If it is reusable approved media, use the asset system.

If it is evidence of a completed decision, use the related QA, approval, or
archive boundary.

If it has multiple roles, keep one authoritative home and add references.

Do not solve a dispute by placing full copies in every plausible folder.

## Enforcement

Contributors must correct a discovered placement error when it is safe to do so.

If correction would break links, release evidence, or active production, create
a documented migration instead of an improvised move.

Reviewers may reject a handoff when ownership, source, status, or release
identity is ambiguous.

Release owners may block upload when the package cannot identify exact approved
deliverables.

Continuity owners may block use of assets whose source relationship is unclear.

Folder compliance is a quality gate, not clerical polish.

## Periodic Review

Review this manual when a recurring placement error appears.

Review it when a new production platform changes custody or export behavior.

Review it when asset volume makes an existing pattern hard to navigate.

Review it after a release, continuity, rights, or recovery failure caused by
unclear storage.

Do not revise it solely to match a short-lived tool interface.

Do not revise it to legitimize an undocumented workaround after the fact.

Any revision must preserve existing top-level folders unless the separate
authority process explicitly approves an actual repository change.

## Final Operating Rules

One source of truth has one owned home.

One file may support many tasks without being copied into many authorities.

One selected output must remain traceable to its source and decision.

One release package must identify the exact public deliverables.

One archive record must explain why active use stopped.

One exception must have an owner, limit, and expiry.

One migration must preserve a recoverable path from old to new.

Folder structure is successful when the correct next action is visible without
depending on memory.

## Guiding Principle

Place work once, preserve its lineage, protect its authority, and make every
handoff safe to continue.
